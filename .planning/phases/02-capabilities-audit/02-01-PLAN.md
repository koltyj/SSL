**Step 6: Feature feasibility probes**

**Soft keys probe (AUDIT-02):**
Attempt the softkey edit flow from 02-RESEARCH.md:
```
ssl> raw 600 01                       # read current keymap name for layer 1
ssl> raw 610 01 <keymap_name_hex>     # open edit session (use name from step above, null-terminated ASCII hex)
ssl> raw 641                          # read keymap size (expect transport=15, softkey=8)
ssl> raw 620 01 00 00                 # read key 1 data
ssl> state                            # check state.softkeys
ssl> raw 680                          # save (close) edit session
```
Document: Can we read keymap names? Open edit sessions? Read key data? What key types exist?

**V-pot / Wheel mode probe (AUDIT-02):**
Wheel mode reads were done in Tier 1 (raw 1060). Now probe CC names:
```
ssl> raw 950 01 00       # CC names list (layer 1, type 0)
ssl> raw 950 02 00       # CC names list (layer 2)
ssl> raw 950 01 01       # CC names list (layer 1, type 1) — may vary
```
Document: What wheel modes exist? What CC names are available? Is CC protocol (type 3) assigned to any layer?

**SuperCue / Auto-Mon probe (AUDIT-02):**
There are no known SuperCue message codes. Test approach:
1. Check if any profiles mention SuperCue or Auto-Mon in their names
2. Probe for undocumented commands near the softkey range (cmd 1100-1200) using raw
3. If nothing found via protocol, document as "hardware-controlled only — not reachable via UDP protocol"

**Split board probe (AUDIT-03):**
```
ssl> layers              # document all 4 layers
ssl> profiles            # document all profiles — look for fader range/group fields
ssl> raw 910 01          # layer 1 protocol
ssl> raw 910 02          # layer 2 protocol
ssl> raw 910 03          # layer 3 protocol
ssl> raw 910 04          # layer 4 protocol
```
If layers 1 and 2 both show active protocols (HUI + MCU):
1. Open Pro Tools and Ableton Live simultaneously
2. Verify both DAWs show connected MIDI surfaces
3. Move faders 1-8 — does Pro Tools respond?
4. Move faders 9-16 — does Ableton respond?
5. If both respond independently: PASS (split board is firmware-native)
6. If only one DAW responds at a time: FAIL (one-active-layer limitation)
7. If partial: document exactly what works

If only 1 layer is active, attempt to activate a second:
```
ssl> setprofile 3 <mcu_profile_name>   # assign MCU profile to layer 3
ssl> layers                             # verify assignment
```
Then repeat the dual-DAW test above.

Fill in all Feature Feasibility sections in CAPABILITIES.md.
